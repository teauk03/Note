# 메모리 할당과 사용 구조

* MySQL에서 사용되는 메모리 공간은 크게 `글로벌`과 `로컬 메모리` 영역으로 구분
* 글로벌 메모리와 로컬 메모리를 구분하는 가장 큰 차이는 MySQL 서버 내에 있는 많는 <U>스레드가 공유해서 사용하는 공간인지</U>에 따라 여부가 다름

<br></br>

# 글로벌 메모리 영역 (Global Memory Area)
* MySQL 서버가 실행될 때 OS에서부터 무조건 한번만 할당 받는 거대한 메모리 공간
* 클라이언트 스레드의 수와 무관하게 생성 되어지며, 모든 스레드(포그라운드, 백그라운드)에 의해 공유되어짐
* 여러 스레드에 의해 공유 되어지기에, 스레드의 동시 접근으로 인한 데이터 오염을 방지하기 위해 동기화 자원(Mutex, Latch)등이 활발하게 일어나는 영역

| 주요 구성 요소명 | 크기 설정 시스템 변수 | 역할 및 특징 |
| :--- | :--- | :--- |
| **InnoDB 버퍼 풀**<br>(InnoDB Buffer Pool) | `innodb_buffer_pool_size` | • 디스크의 데이터 파일과 인덱스 정보를 메모리에 들고 있는 캐시 공간<br>• 읽기 성능 향상 및 쓰기 지연(더티 페이지 버퍼링)의 핵심부 |
| **Redo 로그 버퍼**<br>(Redo Log Buffer) | `innodb_log_buffer_size` | • 트랜잭션의 변경 로그(Redo Log)를 디스크에 쓰기 전 임시로 저장하는 버퍼<br>• WAL 프로토콜의 일기장 역할을 수행하여 장애 복구를 보장 |
| **Binary 로그 캐시/버퍼**<br>(Binary Log Cache) | `binlog_cache_size` | • 디스크의 바이너리 로그 파일에 쓰기 전, 트랜잭션 변경 내역을 임시 저장하는 버퍼<br>• 주로 마스터-슬레이브 복제(Replication) 및 데이터 복구용 일지를 기록 |
| **테이블 캐시**<br>(Table Cache) | `table_open_cache` | • 오픈된 테이블의 메타데이터(구조 정보)나 권한 정보를 보관<br>• 쿼리 실행 시 매번 디스크에서 테이블 구조를 읽는 오버헤드를 방지 |

# 로컬 메모리 영역 (Local / Session Memory Area)
* 클라언트가 접속하여 포그라운드 스레드가 생성되어 특정 쿼리를 실행할 때 필요에 따라 할당되는 메모리 공간 (클라이너트 메모리 영역이라고도 부름)
* 각 클라이언트 **스레드 별**로 <U>독립적으로 할당되어 절대 공유 되어 사용되지 않음</U>
* Too Many Connections 문제(커넥션 수 * 쿼리가 요구하는 버퍼 크기 로 인해 OS 메모리가 고갈 되어 MySQL 프로세스가 강제종료 되는 현상)가 `max_connections` 관리 미숙으로 인해 발생할 수 있음

| 주요 구성 요소명 | 크기 설정 시스템 변수 | 주요 역할 |
| :--- | :--- | :--- |
| **정렬 버퍼**<br>(Sort Buffer) | `sort_buffer_size` | • `ORDER BY`나 `GROUP BY` 같은 정렬 작업을 수행할 때만 반짝 할당되는 공간<br>• 쿼리 실행 완료 시 즉시 OS로 자원이 반환됨 |
| **조인 버퍼**<br>(Join Buffer) | `join_buffer_size` | • 인덱스를 타지 못해 처음부터 끝까지 테이블을 뒤져야 하는 조인(`Full Join`) 시 중간 결과를 담는 공간 |
| **리드 버퍼**<br>(Read Buffer) | `read_buffer_size` | • 테이블을 처음부터 끝까지 순차적으로 읽는(`Sequential Scan`) 작업 시, 디스크 I/O 효율을 높이기 위해 데이터를 임시로 모아두는 버퍼 |
| **네트워크 버퍼**<br>(Net Buffer) | `net_buffer_length` | • 클라이언트와 서버가 데이터를 주고받을 때(접속 회선 통신)
