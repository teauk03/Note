# MySQL 스레딩 구조 

* MySQL 서버는 프로세스 기반이 아니라 스레드 기반으로 작동함
* 이 스레드 기반 서버에서는 스레드의 역할이 크게 `포그라운드 스레드(Foreground Thread)`, `백그라운드 스레드(Background Thread)`로 구분되어 짐
* 동일한 이름의 스레드는 여러 스레드가 동일 작업을 병렬로 처리하는 경우 존재함

## 1.1. 포그라운드 스레드
* **클라이언트 커넥션 연결 및 쿼리 처리**
    * 클라이언트가 MySQL에 접속하는 순간, <U>클라이언트 커넥션당 하나씩 포그라운드 스레드가 할당</U>됨
    * 이 스레드는 클라이언트가 전송한 SQL 쿼리를 직접 수신하고 분석하며 처리함
    * 이 스레드는 <U>데이터를 MySQL의 버퍼나 캐시로부터 가져오며</U>, 버퍼 풀에 데이터가 없는 경우 직접 디스크의 데이터나 인덱스 파일로부터 데이터를 읽어와 버퍼 풀에 적재하고 작업을 처리함 (쓰기 작업은 InnoDB에서 백그라운드 스레드가 전담하지만, 읽기 작업은 포그라운드 스레드가 직접 디스크 I/O를 수행할 수 있음)
* **스레드 캐시를 통한 재활용**
    * 클라이언트의 `커넥션이 종료(Disconnect)`되면 해당 스레드는 즉시 소멸되지 않고 스레드 캐시 영역으로 반환되어 대기 상태로 전환됨
    * 스레드 캐시에 일정 개수 이상의 대기 중인 스레드가 있으면, 스레드 캐시에 넣지 않고 스레드를 종료시켜 일정 개수의 스레드만 스레드 캐시에 존재하게 함 (`thread_cache_size` 시스템 변수로 설정 가능)

## 1.2. 백그라운드 스레드
* 사용자의 요청을 처리하는 중 데이터의 쓰기 작업은 지연될 수 있으나, 읽기 작업은 지연될 수 없다. (지연 쓰기 메커니즘의 기반)
* **MyISAM vs InnoDB**
    * **MyISAM**  
        * 동시성 제어 기술이 부족하여 `테이블 단위 잠금`을 사용
        * 쓰기 작업이 들어오면 테이블을 통째로 잠그고, 포그라운드 스레드가 디스크에 직접 쓸 때까지 모든 I/O 요청을 차단
    * **InnoDB**
        * `행 단위 잠금`과 `비동기 버퍼링(지연 쓰기)` 기술을 사용
        * 쓰기 요청이 오면 메모리(버퍼 풀)에만 빠르게 작성 후, 실제 디스크 I/O 작업은 백그라운드 스레드가 백그라운드에서 비동기로 수행
* **내부 시스템 작업 전담**
    * 클라이언트와 직접 통신하지 않으며, MySQL 내부의 핵심적인 시스템 작업들을 보이지 않는 곳에서 전담하여 처리하는 `스레드들`의 집합
* **여러 작업을 담당하는 스레드들 (InnoDB)**
    - **Insert Buffer를 병합하는 스레드**
        - 디스크에 흩어진 인덱스를 매번 수정하면 성능이 저하되므로, 메모리에 모아두었다가 한 번에 백그라운드에서 정리하고 병합하는 스레드
    - **로그를 디스크로 기록하는 스레드 (Log Writer Thread)**
        - 데이터 유실을 방지하기 위해 `로그 버퍼(Log Buffer)` 메모리에 작성된 변경 이력(Redo Log)을 디스크의 리두 로그 파일로 기록하는 스레드
    - **InnoDB 버퍼 풀의 데이터를 디스크에 기록하는 스레드 (Page Cleaner Thread)**
        - `메모리(버퍼 풀)`에 적혀 있던 변경된 데이터(더티 페이지)를 실제 `디스크 파일(.ibd)`로 비동기적으로 플러시하여 저장하는 스레드 
        - MySQL 5.5 버전부터 `innodb_write_io_threads` 시스템 변수로 스레드의 개수를 설정할 수 있음
    - **데이터를 버퍼로 읽어오는 스레드**
        - 사용자가 필요할 것으로 예측되는 데이터를 미리 디스크에서 버퍼 풀로 올려두는(Read-Ahead) 스레드 
        - MySQL 5.5 버전부터 `innodb_read_io_threads` 시스템 변수로 스레드의 개수를 설정할 수 있음
    - **잠금이나 데드락을 모니터링하는 스레드**
        - 여러 사용자가 동시에 데이터를 수정할 때 자원이 꼬이는 `교착 상태(Deadlock)`를 감시하고, 문제 발생 시 한쪽 트랜잭션을 강제 롤백하여 교착 상태를 해제하는 스레드
